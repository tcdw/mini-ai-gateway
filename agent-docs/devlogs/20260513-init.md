
#### 2. 核心架构与请求处理流程 (`index.ts`)
*   **启动服务**：使用 `Bun.serve({ port: process.env.PORT, fetch: handler })` 启动高性能 HTTP 服务器。
*   **鉴权拦截**：在 `handler` 中，首先校验 `Authorization` Header，如果不等于 `Bearer ${process.env.GATEWAY_API_KEY}`，返回 401。
*   **只处理核心端点**：目前只代理 `POST /v1/chat/completions`。其他路径直接返回 404。
*   **解析与路由**：
    1.  读取 `req.json()` 拿到原始请求体。
    2.  提取 `body.model`。
    3.  查 `routes.json`。**如果找不到模型，直接返回 404 Error (OpenAI 格式的错误 JSON)。**
*   **负载均衡**：如果模型存在，读取对应的 `providers` 数组，根据 `weight` 字段计算总权重，生成一个随机数来选择命中的 Provider。
*   **请求重写 (Remap)**：
    1.  将 `body.model` 替换为选中 Provider 的 `remap` 值。
    2.  根据选中的 Provider，从 `process.env` 中获取对应的 `URL` 和 `KEY`。
*   **透明转发与流式处理**：
    1.  使用 `fetch(PROVIDER_URL + "/chat/completions", { ... })` 发出请求，携带新的 Body 和 `Authorization`。
    2.  拿到 Provider 的 Response 后，**无需缓冲**，直接将这个 Response 对象 `return` 给客户端。由于 Bun 原生支持，这会自动完美处理 SSE (`stream: true`) 和普通 JSON 响应。

#### 3. 失败回退机制 (Graceful Fallback - 可选)
*   **方案**：如果选中的 Provider `fetch` 失败（如网络错误）或者返回 5xx 状态码，自动将其从可用列表中剔除，重新计算剩余 Provider 的权重并重试。如果所有候选 Provider 都失败了，再向客户端返回 500。
